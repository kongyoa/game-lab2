# 校园二手书交易平台需求规格说明书 (SRS)

## 1. 项目概述
本项目旨在为高校学生提供一个便捷、安全、高效的校园二手书交易平台。解决教材昂贵、闲置书籍浪费的问题，促进校园内的资源循环利用。

### 1.1 核心目标
- **便捷发布**：通过ISBN扫描快速发布图书信息。
- **精准匹配**：基于地理位置（校区/宿舍）和价格区间的筛选。
- **安全交易**：集成实名认证、信用评分及第三方支付。
- **实时沟通**：内置即时通讯系统，方便买卖双方沟通。

## 2. 技术架构

### 2.1 前端 (Web & Mobile Web)
- **框架**：Vue.js 3 + TypeScript
- **UI组件库**：Element Plus (PC端) / Vant UI (移动端适配)
- **状态管理**：Pinia
- **路由管理**：Vue Router
- **HTTP客户端**：Axios
- **实时通信**：Socket.IO Client / WebSocket

### 2.2 后端 (RESTful API)
- **框架**：Spring Boot 3.x
- **ORM**：MyBatis-Plus / JPA
- **数据库**：MySQL 8.0
- **缓存**：Redis (用于缓存热点数据、Session共享、防刷限流)
- **消息队列**：RabbitMQ / Kafka (用于订单超时取消、异步通知)
- **搜索引擎**：Elasticsearch (可选，用于高性能全文检索)
- **对象存储**：MinIO / 阿里云OSS (用于存储图书图片、用户头像)

### 2.3 部署与运维
- **容器化**：Docker + Docker Compose
- **反向代理**：Nginx
- **CI/CD**：Jenkins / GitHub Actions

## 3. 功能需求详情

### 3.1 用户模块
| 功能点 | 描述 | 优先级 |
| :--- | :--- | :--- |
| **注册/登录** | 支持学号/手机号注册，集成短信验证码。必须进行**实名认证**（上传学生证或绑定校园卡）。 | P0 |
| **个人中心** | 管理个人信息、收货地址、查看发布的图书、由于订单、收藏夹。 | P0 |
| **信用评分** | 初始分100分。交易成功+1，好评+1，违规/恶意取消订单扣分。低于80分限制发布。 | P1 |
| **消息通知** | 系统通知（订单状态变更）、私信通知。 | P1 |

### 3.2 图书发布与管理模块
| 功能点 | 描述 | 优先级 |
| :--- | :--- | :--- |
| **ISBN识别** | 输入或扫描ISBN码，自动调用API获取图书元数据（书名、作者、出版社、封面）。 | P0 |
| **图书发布** | 补充实物图片（支持多图上传、压缩）、设定价格（原价/售价）、新旧程度、所在校区/宿舍楼。 | P0 |
| **图片处理** | 图片上传前进行前端压缩，后端进行鉴黄/敏感内容检测。 | P1 |
| **图书搜索** | 支持关键字（书名、作者、ISBN）、分类筛选。 | P0 |
| **高级筛选** | 支持按**价格区间**、**地理位置**（距离/校区）、**新旧程度**筛选。 | P1 |

### 3.3 交易与订单模块
| 功能点 | 描述 | 优先级 |
| :--- | :--- | :--- |
| **即时购买** | 锁定库存，生成订单。 | P0 |
| **在线支付** | 集成支付宝/微信支付接口（沙箱环境或个人收款码回调）。 | P0 |
| **交易撮合** | 支持“同校面交”和“快递寄送”两种模式。 | P0 |
| **状态追踪** | 待支付 -> 已支付 -> 待发货/待见面 -> 已发货 -> 确认收货 -> 已完成 -> 评价。 | P0 |
| **售后/退款** | 买家申请退款，卖家审核。平台介入仲裁。 | P2 |
| **防刷单** | 同一IP/设备短时间内频繁下单限制；异常价格监控。 | P1 |

### 3.4 在线沟通模块
| 功能点 | 描述 | 优先级 |
| :--- | :--- | :--- |
| **实时聊天** | 买卖双方通过WebSocket进行文字、图片沟通。 | P0 |
| **交易卡片** | 聊天窗口内可发送当前商品卡片、订单状态卡片，方便快速交互。 | P1 |
| **敏感词过滤** | 聊天内容需经过敏感词过滤系统（自动屏蔽违规词汇）。 | P1 |

### 3.5 评价与反馈
- **双向评价**：交易完成后，买卖双方互评（1-5星及文字评论）。
- **投诉举报**：用户可举报违规图书或恶意用户。

## 4. 数据库设计 (ER核心概念)

### 4.1 核心表结构
1.  **Users (用户表)**
    *   `id` (PK), `username`, `password_hash`, `student_id`, `school_id`, `credit_score`, `avatar_url`, `status`
2.  **Books (图书表)**
    *   `id` (PK), `user_id` (FK), `isbn`, `title`, `author`, `publisher`, `original_price`, `price`, `condition` (新旧), `location_id`, `status` (在售/已预订/已售出), `images` (JSON)
3.  **Orders (订单表)**
    *   `id` (PK), `buyer_id` (FK), `seller_id` (FK), `book_id` (FK), `amount`, `status` (0:待支付, 1:已支付, ...), `payment_method`, `trade_type` (面交/快递), `create_time`, `update_time`
4.  **Messages (消息表)**
    *   `id` (PK), `sender_id`, `receiver_id`, `content`, `msg_type` (text/image/system), `is_read`, `send_time`
5.  **Comments (评价表)**
    *   `id` (PK), `order_id`, `from_user_id`, `to_user_id`, `rating`, `content`, `create_time`

## 5. 非功能性需求

### 5.1 性能要求
- **高并发**：支持至少 500 QPS (Queries Per Second) 的并发访问。
- **响应时间**：API平均响应时间 < 200ms，静态资源加载 < 1s。
- **图片加载**：采用CDN加速或对象存储预处理（缩略图）。

### 5.2 安全性
- **数据加密**：用户密码采用 BCrypt 加密存储；敏感信息（如手机号）脱敏展示。
- **网络安全**：全站启用 HTTPS；API 启用 Rate Limiting (限流) 防止 DDoS。
- **接口安全**：使用 JWT (JSON Web Token) 进行无状态认证；防止 CSRF 和 SQL 注入。

### 5.3 可靠性与测试
- **单元测试**：核心业务逻辑（订单、支付、库存）覆盖率 > 80% (JUnit/Jest)。
- **接口测试**：提供完整的 Postman/Swagger 测试用例。
- **渗透测试**：上线前进行 SQL 注入、XSS、越权访问等安全扫描。
